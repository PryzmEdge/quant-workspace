## Executive Summary

Markdown is a human-readable orchestration substrate — not an execution environment. Its consumers (parsers, runtimes, LLM agents, Pandoc filters, notebook kernels) do the executing. The emerging production architecture is best understood as **deterministic scaffolding around probabilistic reasoning cores**: a pattern where deterministic filesystems, schemas, hooks, and pipelines wrap a nondeterministic LLM reasoner. This is the long-term winning pattern for quant workflows — and for any domain where provenance, reproducibility, auditability, and staged human approval are non-negotiable.

## Part 3 — Deterministic Scaffolding Around Probabilistic Cores

This is the dominant production architecture pattern emerging in 2026, now formally named.

From *"Harness Engineering: The Architecture of Production-Grade AI Systems"* (dev.to, May 2026):

> The math is unforgiving: a 10-step agent chain at 95% per-step accuracy yields 59.9% end-to-end reliability (0.95^10). One deterministic circuit breaker inserted at a critical step resets reliability for that checkpoint to 100%.

### Compiled AI vs. Direct Runtime Inference

A primary mechanism to achieve 100% execution-stage determinism is **Compiled AI**. Under this paradigm, large language models are leveraged exclusively during a **compile-time generation phase** to emit static, executable code artifacts. Once generated, these workflows execute deterministically within pre-validated templates on conventional software infrastructure, completely eliminating runtime model calls and temperature-based variance.

Compiled AI satisfies three core systems properties:

1. **One-time LLM invocation**: The model runs once at generation time, not at transaction time.
2. **Zero-token deterministic execution**: Deployed workflows execute as native code (e.g., Python, Terraform) without further API calls, dropping transaction-stage latency to 4.5ms (a 450x improvement over direct LLM runtimes).
3. **Mandatory multi-stage validation**: Generated code is subjected to automated syntax, execution, and semantic accuracy checks before deployment. This converts potential runtime failures into detectable compile-time blocks, ensuring 100% deployed-workflow reliability.

### The Three Enforcement Layers

| Enforcement Type | Mechanism | Reliability |
|---|---|---|
| `CLAUDE.md` instructions | Advisory context (user-message tier) | Probabilistic |
| Schemas (Pydantic/JSON Schema) | Structured output constraints | High — rejects invalid outputs |
| Hooks (`.claude/settings.json`) | PreToolUse/PostToolUse callbacks | Deterministic — guaranteed execution |

> **Production rule**: Never rely on `CLAUDE.md` for hard constraints in quant workflows. Behavioral rules belong in hooks. Schema contracts belong in Pydantic models. `CLAUDE.md` handles structural navigation and identity — content where probabilistic compliance is acceptable.

## Part 4 — The Memory Hierarchy: LLM Context as Cache

The most important academic paper for this framework in 2026: *"The Missing Memory Hierarchy: Demand Paging for LLM Context Windows"* (arXiv:2603.09023, Tony Mason, March 2026).

Key measurement from the paper: across 857 production sessions and 4.45 million effective input tokens, **21.8% was structural waste** (stale tool results, redundant definitions, dead context).

### Pichay: Virtual Memory Demand Paging for LLMs

To resolve this context bloat, the **Pichay** system interposes as a transparent proxy between the client application and the LLM inference endpoint. Pichay actively manages context tokens by differentiating ephemeral garbage collection from page eviction:

- **Context Eviction**: Stale, bulky files or past tool outputs are surgically evicted from the L1 generation window. In their place, Pichay writes a lightweight, token-efficient retrieval handle (e.g., `<page-ref id="file.md" />`) which acts as a virtual page table entry.
- **Page Fault Detection**: If the model's downstream reasoning paths attempt to reference the paged-out file, the proxy intercepts the message, registers a "page fault", and triggers a tool call to dynamically page (fault) the raw data back into the active L1 window.
- **Inverted Cost Model**: Traditional CPU memory hierarchies minimize page faults because fetching from disk is expensive. For LLMs, this cost model is inverted: keeping active tokens in context is highly expensive due to quadratic O(n²) self-attention scaling, whereas re-fetching (faulting) data is a cheap, one-time transaction cost. Aggressive eviction of cold context is therefore mathematically and financially optimal.

### Aligned Memory Hierarchy

The five-level memory hierarchy maps directly onto classical virtual memory systems:

| Level | OS Analogy | Context Content | Persistence | Source / Reference |
|---|---|---|---|---|
| L1 — Generation Window | CPU L1 cache | Active turn context, system prompts, current tokens | Turn lifetime | arXiv:2603.09023 |
| L2 — Working Set | CPU L2 cache / RAM | Pinned, high-fault-history context & tool state | Session lifetime | arXiv:2603.09023 |
| L3 — Session History | Swap partition | Compressed/compacted turns with declared losses | Session lifetime | arXiv:2603.09023 |
| L4 — Persistent Memory | Local Storage | External, cross-session persistent knowledge | Permanent | arXiv:2603.09023 |
| L5 — Git History † | Archival | Diffs, version state at any commit | Permanent | Extended by author |

ICM/MWP is, at its core, a hierarchical context compression system — it implements L1/L2 separation manually through stage scoping before any automated paging system exists. Token ranges (2,000–8,000 per stage) are not about raw model capacity but about maintaining **cache locality**: keeping only the working set in active context.

## Part 9 — Filesystem vs. Vector Retrieval: Correctly Balanced

### LlamaIndex Benchmark (January 2026)

| Factor | Filesystem Agent | Hybrid RAG |
|---|---|---|
| Accuracy (correctness) | +2 points higher | Baseline |
| Speed | 3.81 sec slower | Faster |
| Scale sweet spot | <100 focused documents | 100–1,000+ documents |
| Context handling | Full file access | Chunked fragments |

### Correct Balanced Architecture

| Corpus Type | Retrieval Strategy |
|---|---|
| Structured/bounded (<100 docs, known schema) | Filesystem-first (grep, ripgrep, cat) |
| Medium heterogeneous (100–1,000 docs) | Hybrid: semantic filter → filesystem precision extract |
| Large unstructured (>1,000 docs, internet-scale) | Vector/hybrid necessary |

### Why Filesystem Wins for Quant's Specific Corpora

Quant research files (schema files, reference docs, config rules — bounded and well-structured) are small, structured, and known in advance. Vector retrieval properties that hurt here:
- **Ambiguity**: "authorization" semantically similar to "authentication" in wrong contexts
- **Reproducibility**: embedding model version changes → different retrieval on same query
- **Provenance**: chunk origin unclear without extra tooling
- **Debuggability**: cannot inspect what was retrieved without specialized tracing

Filesystem retrieval for these corpora: deterministic, inspectable, replayable, auditable.

## Part 12 — Frontier Concept C: CRDT-Style Agent Collaboration

### Eg-walker: Highly Optimized P2P Workspace Convergence

For collaborative, multi-agent workspaces, traditional Operational Transformation (OT) is slow to merge files that have diverged significantly during offline editing, while classical Conflict-Free Replicated Data Types (CRDTs) exhibit heavy memory overheads and slow disk load times. To resolve this, we leverage the **Eg-walker** collaborative text-editing algorithm (Joseph Gentle & Martin Kleppmann, EuroSys 2025, arXiv:2409.14252).

Compared to existing CRDTs, Eg-walker consumes an order of magnitude less memory in the steady state, loads documents from disk orders of magnitude faster, and merges long-running branches significantly faster than traditional OT. Because it requires no central server, it can be deployed directly across decentralized peer-to-peer agent workspaces.

### CodeCRDT: Structured State Consistency

Multi-agent coordination requires a shared state substrate satisfying three consistency properties: (1) observable updates, (2) deterministic convergence, and (3) monotonic progress. To enforce these constraints, we deploy the **CodeCRDT** protocol (arXiv:2510.18893). Under a strong eventual consistency (SEC) model, CodeCRDT introduces a formal TODO-claim protocol which guarantees provable at-most-one-winner safety during parallel task execution, preventing concurrent agents from redundantly claiming or modifying the same workspace files.

## Part 12.5 — Frontier Concept E: The Continuity Layer and DTCM

### Session Amnesia vs. Architectural Continuity

The most critical limitation of current AI agent architectures is the absence of a dedicated state layer that carries forward what a model has come to understand. Sessions end, context windows fill, and standard memory APIs return flat, disconnected facts that the model must constantly re-evaluate and re-interpret from scratch on every read.

The **Continuity Layer** (Samuel Sameer Tanguturi, arXiv:2604.17273) establishes continuity as a system property defined by seven required characteristics, ensuring the model remains connected across time instead of resetting at every interaction:

1. **Model Independence**: Continuity lives below the intelligence layer, maintaining consistent state regardless of the underlying LLM.
2. **Persistence Beyond Session**: The state survives system shutdowns, browser reloads, and device crashes.
3. **Non-Amortized Retrieval**: Replaces simple semantic similarity lookups with deep situational reconstruction.

### Decomposed Trace Convergence Memory (DTCM)

The core storage primitive powering this layer is **Decomposed Trace Convergence Memory (DTCM)**.

- **The Write-Path (Decomposition)**: Rather than saving raw, unstructured text transcripts, DTCM structurally decomposes interaction traces into explicit dimensions: identities, events, commitments, intent, timing, and logistics.
- **The Read-Path (Reconstruction)**: Instead of retrieving a random bag of outdated semantic snippets, DTCM queries these dimensions to reconstruct the current active situation, reflecting updates, cancellations, and real-time state changes.

The formal validation framework for this architecture is the **ATANT benchmark** (arXiv:2604.06710), which evaluates continuity against a 250-story corpus with zero language models in the scoring loop.

## Part 13 — Frontier Concept D: Context Compilation

### Context Engineering Is the Discipline

From Anthropic's September 2025 blog post:

### Claude Context Anxiety and Trimming Strategies

Empirical testing with frontier models reveals **context anxiety**, a behavioral pathology where models (e.g., Claude 3.5/4) wrap up tasks prematurely or truncate logical steps when they sense their context window limits are being approached. To prevent this, context resets and surgical trimming are required:

- **Compaction**: A lossy, whole-transcript operation that flattens conversation history into a condensed summary, resetting the context window while preserving high-fidelity structural facts.
- **Tool-Result Clearing**: A precise, sub-transcript operation that crawls the message history and replaces massive, re-fetchable `tool_result` content blocks (e.g., raw code readouts, database dumps) with lightweight placeholders, preserving the original `tool_use` signature to maintain reasoning continuity while reclaiming up to 90% of active token space.

## Part 18 — Matrix, Vector & Sparse Retrieval Architectures

### Core Insight

A multi-stage Markdown workspace is not just a set of files — it is a corpus with implicit matrix structure. Three distinct layers apply: sparse matrix querying over structured metadata, dense vector retrieval for semantic similarity, and hybrid sparse+dense pipelines for production accuracy.

### Layer 1: Sparse Matrix — Structured Metadata Querying

### Layer 2: KohakuRAG Hierarchical Indexing and Vectorization

Standard flat-chunking strategies slice documents into fixed-length segments, destroying logical structural boundaries. To preserve document syntax for high-precision quant citation tracking, we leverage the **KohakuRAG** framework (1st place, WattBot 2025 Challenge, 0.861 final score).

KohakuRAG represents each markdown file as a four-level tree structure:

Sentence-level embeddings are computed and aggregated bottom-up to parent nodes using length-weighted composition, establishing explicit provenance tracking.

To measure reading order similarity during automated validation pipelines, we use the **Normalized Indel Distance (NID)**:

$$\text{NID}(p, \text{gt}) = \frac{\text{distance}(p, \text{gt})}{\max(|p|, |\text{gt}|)}$$

where `distance` represents the insertion-deletion edit distance between the predicted text sequence and the ground-truth document structure (`gt`).

### Layer 3: Hybrid Sparse + Dense Retrieval

Alpha between 0.5 and 0.7 (weighted toward sparse) is appropriate when queries mix exact identifiers with semantic intent — the dominant pattern in `CONTEXT.md`-driven workspaces.

### Retrieval Architecture Decision Matrix

| Scenario | Best Layer | Implementation |
|---|---|---|
| "Show all AMBER stage outputs" | Sparse matrix (metadata) | pandas + python-frontmatter |
| "Find sections about constraint violations" | Dense vectors | SentenceTransformer + numpy |
| "Find exact artifact ID validation-receipt-2026-05-18" | BM25 sparse | rank-bm25 |
| "Find sections related to this artifact ID AND semantically similar concepts" | Hybrid | alpha ≈ 0.6 weighted BM25 |
| "What is the current workspace health across all stages?" | Sparse matrix (metadata) | pandas `.groupby()` on frontmatter matrix |
| "Which files have unresolved task lists near RED constraint sections?" | Combined sparse matrix + BM25 | Feature matrix filter → BM25 on hits |
